构建实录：Smart Inspector 如何取代"表哥表姐"

2026年1月 · 深度复盘

在信也科技负责菲律宾贷后策略期间，我们面临一个经典痛点：数据太多，决策太少。分析师每天早上要花 1.5 小时从各个看板"搬运"数据，汇总成日报，还要手动查为什么 M1 回收率跌了。

为了解决这个问题，我构建了一个 Smart Inspector (智能巡检) Agent。

核心 Prompt 设计

这个 Agent 的核心不是写代码，而是教会它像一个资深分析师一样思考。以下是我迭代了 20 个版本后的 System Prompt 核心片段：

# Role Definition You are not a data fetcher; you are a Senior Risk Analyst. Do not just report numbers (e.g., "M1 is 88%"). Instead, tell me WHY (e.g., "M1 dropped because Vendor B's contact rate fell by 15% due to the typhoon"). # Analysis Logic 1. Check Baseline (7-day MA). 2. If Deviation > Threshold: - Slice by Channel (SMS/Voice/Robot) - Slice by Vendor (Team A/B/C) 3. Correlate with External Events (e.g., "Typhoon", "Holiday").

通过这段 Prompt，Agent 不再是复读机，它开始输出带有归因的洞察。这直接改变了我们的早会流程：从"汇报数据"变成了"讨论解决方案"。

S0 动态分案：当风控策略遇上产能管理

2025年12月 · 策略架构

传统的分案是静态的：S0 进机器人，S1 进人工。但在菲律宾市场，发薪日（Payday）的案量是平时的 3 倍。如果坚持静态分案，发薪日当天人工团队会崩溃，而平时又会闲置。

我的解决方案是构建一个 动态分案引擎。核心逻辑如下：

def allocate_case(risk_score, current_capacity): """ 根据实时产能动态调整策略阈值 """ # 如果产能告急（如发薪日） if current_capacity.load_factor > 0.9: threshold = 0.8 # 仅高风险进人工 else: threshold = 0.4 # 中低风险也可进人工深挖 return "Manual_Call" if risk_score > threshold else "Robot_SMS"

这个简单的逻辑改变，让我们的 ROI 提升了约 40%。它不再是单纯的风控，而是运营资源的最优配置。
